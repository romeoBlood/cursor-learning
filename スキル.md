# Cursor Skills（スキル）

## 🧭 概要（30秒で思い出す）

- Cursor Skills は==「AI エージェントに専門知識を注入するパッケージ」==。Anthropic が提唱するオープン標準（Agent Skills）に基づく
- ==段階的開示（Progressive Disclosure）==により、必要な時だけコンテキストに読み込まれるためトークン効率が良い
- `.cursor/skills/<name>/SKILL.md` に配置。AI が関連性を自動判断してロードする

## 🧩 要点まとめ

### スキルの基本

| 項目 | 内容 |
|------|------|
| 配置場所 | `.cursor/skills/<name>/SKILL.md` |
| ファイル形式 | マークダウン + YAML frontmatter（`name` と `description` が必須） |
| 読み込み方式 | ==段階的開示（3レベル）== |
| 起動方式 | AI が自動判断 / `disable-model-invocation` で手動のみに制限可能 |
| コンテキスト | ==親エージェントと共有==（サブエージェントのように分離されない） |
| 並列実行 | 不可 |
| 再利用性 | 複数プロジェクトで再利用可能 |

### 段階的開示（Progressive Disclosure）の 3 レベル

> [!important] Zenn 記事の解説に基づく
> スキル最大の特徴。コンテキストウィンドウを効率的に使うため、情報を 3 段階で開示する。

| レベル | 名称 | ロードタイミング | 内容 |
|--------|------|-----------------|------|
| **Level 1** | メタデータ | ==常時ロード== | `name` と `description` のみ。エージェントが全体像を把握し、関連スキルを判断する材料 |
| **Level 2** | 指示 | ==AI がタスクに関連すると判断した時== | `SKILL.md` 全文をコンテキストに読み込み |
| **Level 3** | リソース | ==タスク実行時に必要な場合== | `SKILL.md` 内で参照される追加ファイル・スクリプト・リファレンスを読み込み |

```text
常時             → Level 1: name + description（数トークン）
AI が関連と判断   → Level 2: SKILL.md 全文（数百〜数千トークン）
実行に必要な時    → Level 3: 追加ファイル・スクリプト等
```

==Rules（always タイプ）は常に全文ロードされるのに対し、Skills は Level 1 のメタデータだけが常時ロードされるため、トークン消費が大幅に少ない。==

### SKILL.md の構文

```yaml
---
name: スキル名
description: "スキルの説明（AI が関連性を判断する材料になる）"
---

# スキルのタイトル

## When to Use
- どんな場面で使うか

## Steps
1. 手順1
2. 手順2

## Notes
- 補足事項
```

- `name`: スキルの識別名
- `description`: ==AI が自動呼び出しの判断に使う説明文==。明確に書くほど適切にトリガーされる

### スキルの呼び出し制御

| 方式 | 説明 |
|------|------|
| AI 自動判断（デフォルト） | エージェントがタスクの関連性を判断して Level 2 をロード |
| `disable-model-invocation` | AI による自動呼び出しを無効化。手動（`/スキル名`）でのみ起動 |

### Rules / Skills / Commands の使い分け

> [!tip] Zenn 記事からの引用
> - **Rules** は「常に守るべきルール」→ 常時ロードでトークン消費
> - **Skills** は「必要な時に呼ばれる専門知識」→ 段階的開示でトークン節約
> - **Commands** は「ユーザーが明示的に起動するショートカット」→ 手動のみ

==タスク固有の長い知識を Rules に書くと常時トークンを消費する。Skills に分離すれば、必要な時だけロードされて効率的。==

> **ポイント**
> ==スキルの本質は「段階的開示による効率的な専門知識注入」。Rules と違い、必要な時だけ AI が自動でロードするため、大量の知識を低コストで保持できる。==

## 🧪 具体例・検証

### デプロイスキルの例

```yaml
# .cursor/skills/deploy-app/SKILL.md
---
name: deploy-app
description: "アプリケーションのデプロイ手順。デプロイ関連の質問時に使用"
---

# Deploy App

## When to Use
- 本番・ステージング環境へのデプロイ時
- デプロイスクリプトの修正時

## Steps
1. テストを実行: `npm test`
2. ビルド: `npm run build`
3. デプロイ: `scripts/deploy.sh <environment>`

## Notes
- 本番デプロイ前に必ずステージングで確認
```

### Vercel の React Best Practices スキル（実例）

> [!quote] Zenn 記事の紹介
> 10年以上の React/Next.js 最適化知識を含む 40 以上のルールを 8 カテゴリに分類。非同期ウォーターフォール排除、バンドルサイズ最適化、再レンダリング最適化などを網羅。

→ このような大量の知識を Rules に書くと常時ロードされてトークンを圧迫するが、Skills なら関連するタスクの時だけ Level 2 でロードされる。

### コード実行を含むスキル

スキルは Python スクリプト等を Level 3 リソースとして含むことができる。LLM が不得意なタスク（PDF フォーム抽出、リスト処理等）をスクリプトで補完する用途。

### Claude Code Skills との比較

| 観点 | Cursor Skills | Claude Code Skills |
|------|--------------|-------------------|
| 配置場所 | `.cursor/skills/<name>/SKILL.md` | `.claude/skills/<name>/SKILL.md` |
| 読み込み方式 | ==段階的開示（Level 1→2→3）== | プロンプト展開（全文ロード） |
| 自動呼び出し | AI が関連性を判断して自動ロード | AI が必要に応じて自動呼び出し可能 |
| 手動呼び出し | `disable-model-invocation` で制限可能 | `/スキル名` で手動起動 |
| 実行制御 | 基本的になし | ツール制限、モデル指定、サブエージェント指定 |
| 引数 | なし | `$ARGUMENTS` / `$0` で位置引数対応 |
| 互換性 | ==`.claude/skills/` も検出可能== | Cursor Skills フォーマットも利用可 |

## ⚠️ つまずきポイント・アンチパターン

- ==**Rules と Skills の違いを理解せずに全部 Rules に書く**== → Rules は常時ロードでトークンを消費。タスク固有の知識は Skills に分離すべき
- **description を曖昧に書く** → AI が自動呼び出しの判断に使うので、「いつ使うか」を具体的に書く
- **スキルを Cursor と Claude Code で別々に書く** → `.claude/skills/` に配置すれば Cursor も検出可能。共通スキルは 1 箇所に統一すべき
- **小さなルールまでスキルにする** → 数行程度の制約は Rules（always）の方がシンプル。スキルは「まとまった専門知識」向け
- **Level 3 リソースを大量に含める** → 必要時にロードされるとはいえ、巨大なファイルはコンテキストを圧迫する

---

## 🔁 Claude Code での活用方法

- Claude Code の Skills も同じオープン標準（Agent Skills）に基づく
- `.claude/skills/<name>/SKILL.md` に配置すれば Cursor からも参照可能
- Claude Code では段階的開示ではなく全文ロードだが、AI が必要に応じて呼び出す点は同様
- 両ツール併用時は `.claude/skills/` に統一配置が推奨

例：

```text
両ツールで共通スキルを使う場合:
  .claude/skills/deploy-app/SKILL.md に配置
  → Claude Code: /deploy-app で起動（$ARGUMENTS で引数も可）
  → Cursor: AI が自動検出、または /deploy-app で手動起動
```
