# Cursor Skills vs サブエージェント

## 🧭 概要（30秒で思い出す）

- ==スキルは「親エージェントに専門知識を注入」、サブエージェントは「独立したコンテキストで並列作業」==
- 短時間・単一目的のタスクはスキル、長時間・並列・コンテキスト分離が必要ならサブエージェント
- サブエージェントは各々独立したコンテキストウィンドウを消費するため、コストが高い

## 🧩 要点まとめ

### 根本的な違い

> [!important] Zenn 記事の比較に基づく

| 観点 | スキル | サブエージェント |
|------|--------|-----------------|
| **本質** | ==専門知識パック== | ==独立した作業者== |
| **コンテキスト** | ==親エージェントと共有== | ==独立（分離）== |
| **並列実行** | ==不可== | ==可能（最大10並列）== |
| **バックグラウンド** | 不可 | 対応（`background: true`） |
| **再利用性** | 複数プロジェクト可 | 複数プロジェクト可 |
| **適用タイミング** | AI 自動判断 or 手動 | AI 自動判断 or 手動 |
| **トークンコスト** | ==低い（親と共有）== | ==高い（独立ウィンドウ × 並列数）== |
| **読み取り専用** | — | `readonly: true` で制限可能 |
| **モデル指定** | — | `model` パラメータ対応 |
| **ネスト** | — | 対応（サブエージェント→サブエージェント） |

### スキルを選ぶべき場合

> [!quote] Zenn 記事から

- ==単一目的の短時間タスク==（例：チェンジログ生成、コードフォーマット）
- ==汎用的な専門知識の注入==（例：GraphQL 設計、Kubernetes 運用、React ベストプラクティス）
- コンテキスト分離が不要
- 業界標準のベストプラクティスを共有したい
- 親エージェントの会話の流れの中で知識を使いたい

### サブエージェントを選ぶべき場合

> [!quote] Zenn 記事から

- ==長時間のリサーチタスク==（コンテキスト分離が必要）
- ==複数ワークストリームの並行実行==（例：テスト実行 + コード調査 + ドキュメント更新を同時に）
- 多段階のステップを要する専門作業
- ==作業結果を独立した形で検証したい==
- ==メインエージェントのコンテキスト汚染を避けたい==

### パフォーマンス考慮事項

> [!warning] Zenn 記事の注意点
> サブエージェントは各々独立したコンテキストウィンドウを消費するため、==5つを並列実行するとおおよそ5倍のトークンを使用する==。シンプルで短時間のタスクではメインエージェント（+スキル）の方が高速かつ低コスト。

### 判断フローチャート

```text
タスクが来た
  ├─ 短時間で完結する？ → Yes → 専門知識が必要？
  │                              ├─ Yes → スキル
  │                              └─ No  → Rules / AGENTS.md / 直接指示
  │
  └─ 長時間 or 並列が必要？ → Yes → コンテキスト分離が必要？
                                     ├─ Yes → サブエージェント
                                     └─ No  → スキル（ただし並列不可に注意）
```

> **ポイント**
> ==「知識の注入」ならスキル、「作業の委任」ならサブエージェント。迷ったら「親エージェントのコンテキストを汚したくないか？」で判断する。==

## 🧪 具体例・検証

### スキルが適切な例

```text
場面: TypeScript プロジェクトでコードレビュー
→ スキル「typescript-review」に Review チェックリストを定義
→ AI がコードレビュータスクを検知して自動ロード
→ 親エージェントの会話の中で知識を適用
→ コンテキスト分離不要、短時間で完結
```

### サブエージェントが適切な例

```text
場面: 大規模リファクタリングの事前調査
→ サブエージェント「researcher」（readonly: true）を3つ並列起動
  - Agent 1: 影響を受けるファイルの洗い出し
  - Agent 2: テストカバレッジの確認
  - Agent 3: 依存関係の調査
→ 各エージェントが独立コンテキストで調査
→ 結果を親エージェントに統合
→ 親のコンテキストは調査の大量ログで汚れない
```

### サブエージェント定義の例

```yaml
# .cursor/agents/researcher.md
---
name: researcher
description: "コードベースの調査・分析を行うリサーチャー"
model: fast
readonly: true
background: false
---

コードベースを調査し、関連するファイル・関数・パターンを特定してください。
変更は行わず、調査結果のみを報告してください。
```

### Claude Code との対応関係

| Cursor | Claude Code | 備考 |
|--------|-------------|------|
| スキル | Skills（`.claude/skills/`） | 両者ともオープン標準。相互検出可能 |
| サブエージェント | Explore / Plan / General-purpose | Claude Code はデフォルト3種 + カスタム |
| サブエージェント（`readonly`） | Explore エージェント | 読み取り専用の調査 |
| サブエージェント（`background`） | バックグラウンドエージェント | 非同期実行 |
| — | worktree 分離 | ==Claude Code 固有。Cursor にはない== |

## 📖 参考知識（Claude Code との比較）

Cursor と Claude Code は「スキル」「サブエージェント」という同じ用語を使うが、仕組みが異なる。混同しないよう整理する。

| 用語 | Cursor での意味 | Claude Code での意味 |
|------|----------------|---------------------|
| スキル | 段階的開示の専門知識パック（Level 1→2→3 で必要時のみロード） | プロンプト展開型のワークフロー定義（全文ロード、ツール・モデル制御あり） |
| サブエージェント | `.cursor/agents/` で定義、最大10並列、`readonly` / `background` 対応 | Explore / Plan / General-purpose 等のデフォルト3種 + `.claude/agents/` でカスタム定義 |

==名前は同じでも仕組みが異なる点に注意。==

## ⚠️ つまずきポイント・アンチパターン

- ==**何でもサブエージェントにする**== → 短時間タスクは親エージェント + スキルの方が高速・低コスト。サブエージェントは起動オーバーヘッドがある
- ==**コンテキスト分離の必要性を考えずに選ぶ**== → 分離が不要なのにサブエージェントを使うとトークンの無駄
- **スキルに長時間タスクを任せる** → スキルは親のコンテキストを共有するため、長い出力で親のコンテキストが圧迫される
- **サブエージェントの並列数を最大にする** → 10並列可能だが、トークンコストも比例して増加。必要最小限に
- ==**スキルとサブエージェントを混同する**== → スキル＝知識の注入（何を知っているか）、サブエージェント＝作業の委任（何をさせるか）

## 🔁 Claude Code での活用方法

- Claude Code でも同様の使い分けが有効：
  - 専門知識の注入 → `.claude/skills/` にスキル定義
  - 独立した調査・並列作業 → Explore / Plan / General-purpose サブエージェント
- Claude Code 固有の強み：`isolation: worktree` でファイルシステムレベルの分離が可能

例：

```text
Claude Code での使い分け:
  スキル的用途:
    .claude/skills/deploy-app/SKILL.md → デプロイ手順の専門知識注入

  サブエージェント的用途:
    Agent(Explore) → コードベース調査（読み取り専用・高速）
    Agent(Plan) → 設計方針策定
    Agent(General-purpose) → フルツールアクセスの汎用作業
    isolation: worktree → ファイル変更を安全に分離
```
