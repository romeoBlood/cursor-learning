# Cursorにおけるモデル選択とルール設定の仕組み

## 🧭 概要（30秒で思い出す）

- Cursorでは複数のLLMモデル（Claude, GPT-4o, Gemini等）を選択できるが、`.cursor/rules/` の設定はモデル非依存で共通読み込みされる
- モデル別にルール・スキル・スラッシュコマンドを自動切り替えする公式機能は存在しない
- Claude Codeとはルール管理のアーキテクチャが根本的に異なる

## 🧩 要点まとめ

- `.cursor/rules/` 配下の `.mdc` ファイルはどのモデルを選んでも**共通で読み込まれる**
- モデルごとに異なるルールファイルを自動ロードする仕組みは**ない**
- ルール内で「もしあなたがClaude系なら〜」と条件分岐を書くことは可能だが、モデルの自己認識に依存するため不確実

> **ポイント**
> Cursorのルール設定は「プロジェクト単位の共通指示」であり、モデル単位の指示ではない。
> モデルを切り替えても同じルールが適用される。

---

## 🧪 具体例・検証

```text
.cursor/
  rules/
    general.mdc        ← 全モデル共通で読み込まれる
    coding-style.mdc   ← 全モデル共通で読み込まれる
```

- `.cursor/rules/claude-specific.mdc` のようなファイルを置いても、GPT-4o選択時にも読み込まれる
- ファイル名やディレクトリ構造でモデル別フィルタリングはされない

---

## ⚠️ つまずきポイント・アンチパターン

- **モデル別にルールを分けて管理できると思い込む** → 実際には全て共通読み込み
- **ルール内で「あなたはClaudeですか？」と条件分岐** → モデルによっては自分のモデル名を正確に返さない場合がある。確実ではない
- **モデル固有の機能（Artifacts, Canvas等）を前提としたルール** → 別モデル選択時にエラーや無視の原因になる

---

## 🔁 Claude Code との違い・比較

| 項目 | Cursor | Claude Code |
|------|--------|-------------|
| モデル選択 | UI上で複数モデルから選択可能 | 固定（Claude系のみ）。`/model` で切替可能 |
| ルール設定場所 | `.cursor/rules/*.mdc` | `CLAUDE.md`（プロジェクトルート）、`.claude/` 配下 |
| モデル別ルール | 非対応（共通読み込み） | そもそもClaude専用なので不要 |
| ルール適用の確実性 | モデルにより解釈精度が異なる | Claude専用のため高い整合性 |
| スラッシュコマンド | カスタム定義不可（組み込みのみ） | スキル（skill）として拡張可能 |
| メモリ機能 | なし | `.claude/` 配下に永続メモリあり |
| ルールのスコープ | プロジェクト / グローバル | プロジェクト / ユーザー / セッション |

### 重要な違い

- **Cursorは「汎用IDE × 複数LLM」** のため、ルールはモデル非依存の共通指示として設計されている
- **Claude Codeは「Claude専用CLI」** のため、`CLAUDE.md` の指示がClaudeに最適化された形で確実に反映される
- Claude Codeでは `CLAUDE.md` に書いた指示はシステムプロンプトとして注入されるため、ルール遵守の精度が高い
- Cursorのルールは各モデルのプロンプト処理に依存するため、モデルによって解釈のブレが生じうる
